# Output Specification

The scanner should produce machine-readable files first and a maintainer-facing
summary second.

## crypto_inventory.csv

One row per evidence item.

Columns:

- `repo_id`
- `file_path`
- `line`
- `category`
- `primitive_or_protocol`
- `matched_text`
- `context_kind`
- `confidence`
- `runtime_hint`
- `source_kind`
- `notes`

Example categories:

- `rsa`
- `ecdsa`
- `secp256k1`
- `ed25519`
- `jwt`
- `tls`
- `pqc_signal`
- `crypto_library`

Example context kinds:

- `runtime`
- `manifest`
- `lockfile`
- `docs`
- `test`
- `example`
- `generated`
- `vendor`
- `tooling`
- `catalog`
- `fixture`
- `unknown`

Dependency manifests and lockfiles are dependency inventory context. Runtime
source imports of crypto libraries are marked through `source_kind` as
`import_dependency_usage`; this is still passive evidence, not a vulnerability
claim.

- `edge_kinds`
- `edge_basis`

## crypto_findings.sarif

SARIF 2.1.0 export for code review tools. Messages must remain passive:

- static review signal only;
- evidence grade and promotion requirements included;
- not proof of exploitability;
- no live reachability or vulnerability claim.

## crypto_cbom.json

CBOM-style passive cryptographic inventory export. This is not a formal proof of
runtime use. It preserves contexts, evidence grades, source kinds, verdicts, and
sample evidence locations for crypto/PQC migration planning.

## qday_risk_report.json

Required top-level keys:

- `repo`
- `scan_timestamp`
- `scanner_version`
- `claim_boundary`
- `summary`
- `findings`
- `review_paths`
- `evidence`
- `verifier`
- `limitations`

`review_paths` ranks files for manual migration review. It is a passive
prioritization aid, not proof that a path is reachable or vulnerable. Each row
includes:

- `file_path`
- `rank_score`
- `contexts`
- `edge_kinds`
- `edge_basis`
- `primitive_or_protocols`
- `verdicts`
- `evidence_count`
- `claim_state`
- `review_reason`

## qday_risk_report.md

Human-readable report sections:

1. Scope
2. Summary
3. Top Findings
4. Top Review Paths
5. Evidence
6. What This Does Not Prove
7. Recommended Review Steps
8. Claim Discipline
9. Reproduction Command

Required wording:

> This is a passive static review of repository contents. It is not proof of
> exploitability and does not claim that any production system is vulnerable.

## internal_review_note.md

Short local triage note generated by the scanner before verifier approval. It is
not maintainer-facing and must say it is not verifier-approved.

Rules:

- Maximum 250 words by default.
- Include only scanner review targets, not verifier-approved claims.
- Include "not verifier-approved" and "do not send".
- Use "review" and "migration risk", not "hack" or "exploit".
- Mention exact file paths and lines.

## verified_maintainer_note.md

Short maintainer-facing note generated only by `verify_crypto_radar.py` after
line evidence and claim wording checks.

Rules:

- Include only verifier-approved findings.
- Use "review" and "migration risk", not "hack" or "exploit".
- Mention exact file paths and lines.
- Offer diff/PR only if patch verifier passed.
